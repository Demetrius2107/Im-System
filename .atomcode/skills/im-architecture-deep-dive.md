# IM 架构深度设计 — 自检清单与执行框架

> 适用场景：对任意 IM 核心模块进行系统性深度设计（五层法）
> 调用方式：`use_skill("im-architecture-deep-dive", "module=消息可靠投递")`
> 基于 DDD 六边形架构，与 Vela 项目结构对齐

---

## 一、核心认知

### 1.1 项目层级定义

| 层级 | 代码量 | 核心特征 | 目标 |
|------|--------|----------|------|
| Demo级 | 几千行 | 功能跑通 | ✅ 当前状态 |
| 项目级 | 1-3万行 | 功能完整，基本架构 | 短期目标 |
| 生产级 | 5-10万行 | 能扛流量，有容灾降级 | 中期目标 |
| 大厂级 | 几十万行+ | 百万并发，全球多活，成本可控 | 长期目标 |

### 1.2 "深度"的五个维度

```
第一层：能跑通（主流程）
    ↓
第二层：能扛异常（边界条件全覆盖）
    ↓
第三层：能扛并发（性能优化）
    ↓
第四层：能扛故障（容灾降级自愈）
    ↓
第五层：能扛面试（决策论证，能讲30分钟）
```

### 1.3 代码质量演进

| 阶段 | 特征 | 目标 |
|------|------|------|
| 现在 | 满屏 if-else | 理解问题 |
| 下一步 | 策略模式 + 管道 + 状态机 | 隔离复杂度 |
| 最终 | 配置化 + 规则引擎 + 数据驱动 | 运营自助 |

## 二、单点爆破执行框架

### 2.1 选择爆破点（推荐优先级）

**第一梯队（面试必问，必须深挖）：**
1. 消息可靠投递（ACK + 去重 + 有序）
2. 群聊消息扩散（写扩散 vs 读扩散）
3. 消息存储策略（分表分库 + 冷热分离）

**第二梯队（面试加分）：**
4. 离线消息（Redis ZSet 容量与降级）
5. 已读回执（单聊 + 群聊聚合）
6. 消息推送（多通道 + 兜底策略）

**第三梯队（架构深度）：**
7. 消息撤回（连锁场景 + 时间窗口）
8. 文件上传（分片 + 秒传 + 断点续传）
9. 多端同步（增量同步 + 消息漫游）

### 2.2 每一层的检查清单

#### 第一层：主流程跑通
- [ ] 正常场景的功能代码是否完成？
- [ ] 是否有基本的单元测试？
- [ ] 是否能通过手动测试验证？

#### 第二层：异常边界全覆盖

对每个步骤问自己：
- [ ] 如果这一步失败了，用户看到什么？（用户体验）
- [ ] 如果这一步失败了，数据会不一致吗？（数据一致性）
- [ ] 如果这一步失败了，怎么知道？（监控告警）
- [ ] 如果这一步失败了，怎么自动恢复？（自愈机制）
- [ ] 如果这一步失败了，人工怎么介入？（运维工具）

**具体异常场景清单：**

| 场景 | 检查项 |
|------|--------|
| 网络超时 | 重试几次？退避策略？最终失败怎么展示？ |
| 服务挂了 | Redis挂？MQ挂？DB挂？APNs挂？降级策略？ |
| 数据超限 | 离线消息10万条？群成员1万人？文件10GB？ |
| 并发冲突 | 同一条消息同时被撤回和转发？ |
| 时间边界 | 第119秒发撤回，服务端收到时121秒？ |
| 多端冲突 | 手机发消息，电脑同时登录，状态怎么同步？ |

#### 第三层：高并发性能
- [ ] 瞬间高并发怎么削峰？（MQ？令牌桶？）
- [ ] 同一会话消息怎么保序？（分区策略？）
- [ ] 热点数据怎么打散？（一致性哈希？）
- [ ] 推送怎么批量聚合？（时间窗口？）
- [ ] 连接数怎么管理？（连接池？限流？）

#### 第四层：容灾降级自愈
- [ ] 组件故障检测机制？（心跳？健康检查？）
- [ ] 故障自动切换？（主从切换？多活？）
- [ ] 降级开关？（手动？自动？）
- [ ] 熔断器策略？（阈值？恢复策略？）
- [ ] 数据补偿机制？（定时扫描？异步修复？）

#### 第五层：决策论证（面试层）

对每个技术选型，必须能回答：
- [ ] 候选方案有哪些？（至少3个）
- [ ] 你的场景是什么？（具体需求）
- [ ] 排除过程是什么？（为什么不用其他的）
- [ ] 最终选择的代价是什么？（放弃了什么）
- [ ] 什么条件下会换方案？（边界条件）

## 三、代码结构演进指南

### 3.1 从 if-else 到策略模式

适用于：消息类型分发、发送校验策略、推送通道选择

```java
// Step 1: 抽接口
interface Strategy { 
    boolean supports(Context ctx); 
    Result execute(Context ctx); 
}

// Step 2: 每个分支变独立类
class AStrategy implements Strategy { ... }
class BStrategy implements Strategy { ... }

// Step 3: 自动注册路由
@Autowired List<Strategy> strategies;
for (Strategy s : strategies) {
    if (s.supports(ctx)) return s.execute(ctx);
}

// Step 4: 业务代码只剩一行
router.route(ctx);
```

### 3.2 多步骤流程用管道

适用于：消息发送流程（校验 → 去重 → 存储 → 推送 → 回调）

```java
// Step 1: 定义管道节点
interface PipeNode<T> { 
    void process(T ctx, PipeChain<T> chain); 
}

// Step 2: 每个步骤独立实现
class LimitNode implements PipeNode<MsgCtx> { 
    void process(MsgCtx ctx, PipeChain<MsgCtx> chain) {
        // 限流逻辑
        chain.next(ctx); // 继续
    }
}
class FilterNode implements PipeNode<MsgCtx> { ... chain.next(ctx); }

// Step 3: 配置组装
PipeChain.of(limitNode, filterNode, persistNode, pushNode);

// Step 4: 业务代码一行
chain.process(ctx);
```

### 3.3 复杂状态用状态机

适用于：消息状态（发送中 → 已送达 → 已读 → 已撤回）、好友请求状态

```java
// Step 1: 定义状态和事件
enum State { SENDING, DELIVERED, READ, RECALLED }
enum Event { SEND_OK, ACK, READ, RECALL }

// Step 2: 配置流转规则
SENDING --SEND_OK--> DELIVERED --ACK--> READ --RECALL--> RECALLED

// Step 3: 每个流转挂 action 和 guard
.action(notifyAction()).guard(within2MinGuard())

// Step 4: 业务代码触发事件
stateMachine.sendEvent(RECALL, messageId);
```

### 3.4 最终目标：配置化

```
把策略条件外置：
  - YAML 配置文件
  - 数据库规则表
  - 规则引擎（如 Drools）

目标：运营改配置上线，不需要发版
```

## 四、核心模块深度检查清单

### 4.1 消息可靠投递

| 层次 | 检查项 | 状态 |
|------|--------|------|
| L1 | 客户端发 → 服务端落库 → 推送给接收方 → 接收方ACK → 标记已送达 | ☐ |
| L2 | 网络超时重试？ACK丢失重推？DB写入失败补偿？ | ☐ |
| L2 | 消息去重机制？（messageId幂等？） | ☐ |
| L2 | 客户端时间不同步，超时判断以谁为准？ | ☐ |
| L3 | 高并发削峰？（MQ异步落库？） | ☐ |
| L3 | 同会话保序？（分区策略？） | ☐ |
| L3 | 批量推送聚合？（时间窗口？） | ☐ |
| L4 | Redis挂了降级写MySQL？ | ☐ |
| L4 | MQ挂了降级直接写DB？ | ☐ |
| L4 | APNs挂了切换厂商通道？ | ☐ |
| L5 | 为什么选RocketMQ而不是Kafka？（事务消息？延迟消息？） | ☐ |
| L5 | 为什么重试3次？（收益递减点？可调参？） | ☐ |
| L5 | 全球多活sequence怎么保证全局有序？（Snowflake加机房ID？向量时钟？） | ☐ |

### 4.2 群聊消息扩散（写扩散 vs 读扩散）

| 检查项 | 状态 |
|--------|------|
| 小群（<200人）用写扩散，每个成员inbox写一条？ | ☐ |
| 大群（>1000人）用读扩散，只写一条到群消息表？ | ☐ |
| 超大群（10000人）怎么实现？（读扩散 + 缓存预热？） | ☐ |
| 写扩散的存储成本怎么算？（200人 × 1条消息 × 1年 = ?） | ☐ |
| 读扩散的查询性能怎么保证？（索引优化？缓存？） | ☐ |
| 群成员列表变更时，扩散策略怎么切换？ | ☐ |
| @所有人的消息，是否走特殊扩散逻辑？ | ☐ |

### 4.3 消息存储策略

| 检查项 | 状态 |
|--------|------|
| 消息体和消息索引为什么分两张表？（读写分离？） | ☐ |
| 分片键选什么？（按会话ID+时间？还是owner_id？） | ☐ |
| 大群热点怎么打散？（一致性哈希？） | ☐ |
| 单分片数据量上限？（10亿条？怎么扩容？） | ☐ |
| 冷热分离策略？（7天热数据SSD？3个月冷数据S3？） | ☐ |
| 归档数据怎么查询？（异步加载？用户体验？） | ☐ |
| 存储成本怎么算？（1亿用户 × 10条/天 × 1年 = ? PB?） | ☐ |

### 4.4 离线消息（Redis ZSet）

| 检查项 | 状态 |
|--------|------|
| ZSet容量上限？（单用户最多500条？） | ☐ |
| 超出上限怎么降级？（落DB？查询时合并？） | ☐ |
| Redis主从切换时丢数据怎么办？（双写MySQL？） | ☐ |
| 用户同时登录10台设备，都拉取怎么优化？（缓存？） | ☐ |
| 拉取后是否立即删除？（网络超时重复拉取怎么幂等？） | ☐ |
| Redis挂了怎么降级？（直接从DB拉取？性能牺牲？） | ☐ |

### 4.5 已读回执

| 检查项 | 状态 |
|--------|------|
| 单聊已读回执实时推送？ | ☐ |
| 群聊1000人，已读回执怎么聚合？（每10人聚合一次？） | ☐ |
| 已读回执存储成本？（1000人群 × 1000条消息 = 100万条记录？） | ☐ |
| 用户离线，上线后怎么补发已读状态？ | ☐ |
| "免打扰但@我除外"时，已读回执怎么算？ | ☐ |

### 4.6 消息撤回

| 检查项 | 状态 |
|--------|------|
| 时间窗口以服务端为准还是客户端？（时钟同步？） | ☐ |
| 第119秒发请求，服务端收到时121秒，怎么处理？ | ☐ |
| 消息已被转发，连锁撤回还是只撤原始？ | ☐ |
| 接收方正在引用这条消息打字，撤回后输入框怎么办？ | ☐ |
| 用户离线，上线后看到撤回提示但本来没收到，体验怎么设计？ | ☐ |
| 弱网环境下撤回流言丢失，接收方还能看到原消息？ | ☐ |
| 1秒内100条撤回请求，怎么防刷屏？ | ☐ |

### 4.7 消息推送（多通道）

| 检查项 | 状态 |
|--------|------|
| 在线优先（长连接推送）？ | ☐ |
| 离线转APNs/厂商通道？ | ☐ |
| 多通道优先级队列？（APNs > 华为 > 小米 > FCM？） | ☐ |
| 通道失败自动切换？（重试3次后降级？） | ☐ |
| 聚合推送策略？（100ms窗口合并？） | ☐ |
| 静默推送？（仅更新角标？） | ☐ |
| 免打扰时段？（夜间/会议/自定义？） | ☐ |

## 七、关键认知提醒

```
1. 代码量不是目标，"想到第五层"才是
2. 不是模块多，是每个模块的场景覆盖全
3. 不是学更多技术，是把一个技术用到极致
4. 大厂代码不是不用if-else，是if-else被封装在框架层
5. 异常处理代码应该比正常逻辑多
6. 每个技术选型都要能回答"为什么选A不选B"
7. 配置化是终极目标：运营改配置，不用发版
```
