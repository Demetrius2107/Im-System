## 二、逐层差距分析

### 2.1 L2 异常边界 — 当前状态：几乎空白 ❌

| 异常场景 | 当前实现 | 问题 |
|----------|----------|------|
| 网络超时重试 | 无 | 消息发送失败直接返回，没有重试机制 |
| 服务降级 | 无 | Redis/MQ/DB 挂了没有任何降级处理 |
| 离线消息超限 | 简单删除最旧消息 | 没有降级到 DB 或通知用户 |
| 并发冲突 | 无 | 同一条消息同时被撤回和转发未处理 |
| 时间边界 | 硬编码 120s | 客户端与服务端时钟不同步，121秒到达请求直接拒绝 |
| 多端冲突 | 无 | 多端同时操作同一会话未处理 |
| ACK 丢失 | 无重推机制 | 接收方 ACK 丢失，发送方认为对方没收到 |
| DB 写入失败 | 无补偿 | 消息存储失败无补偿机制 |
| 参数校验 | 部分 @Validated | 部分边界值未校验 |

### 2.2 L3 高并发性能 — 当前状态：无优化 ❌

| 性能维度 | 当前实现 | 问题 |
|----------|----------|------|
| 线程池 | 固定 8 核心线程池 | 队列 1000，无拒绝策略，高并发直接丢消息 |
| 消息削峰 | 无 | 网关直接同步调业务服务，没有 MQ 削峰 |
| 群聊分发 | 遍历所有成员逐个发 | 1000 人群，循环 1000 次发送，性能差 |
| 批量推送 | 无 | 每次推送一条消息，没有时间窗口聚合 |
| 同会话保序 | 无 | 多线程并发处理同一会话消息可能乱序 |
| 热点打散 | 无 | 大群消息是热点，没有打散策略 |
| 连接数管理 | 无 | 无连接数统计和限流 |
| 缓存 | 无 L1 本地缓存 | 所有查询都走 Redis，高频访问压力大 |

### 2.3 L4 容灾降级 — 当前状态：无 ❌

| 组件 | 当前实现 | 问题 |
|------|----------|------|
| Redis 挂 | 无降级 | 离线消息、Session、序列全部不可用 |
| MQ 挂 | 直接报错 | 消息存储无法异步落库 |
| DB 挂 | 直接报错 | 业务完全不可用 |
| ZooKeeper 挂 | 无降级 | 服务发现不可用 |
| 网关节点挂 | 无自动切换 | 客户端连接断开 |
| 熔断器 | 无 | 下游故障会级联上游 |
| 健康检查 | 无 | 无法感知组件健康状态 |
| 数据补偿 | 无 | 无定时扫描/异步修复机制 |

### 2.4 L5 决策论证 — 当前状态：文档缺失 ❌

| 决策项 | 当前状态 |
|--------|----------|
| 为什么选 RabbitMQ 不是 Kafka/RocketMQ？ | 文档未记录 |

## 三、代码结构问题

### 3.1 需要重构的 if-else 密集区

| 文件 | 问题 | 建议模式 |
|------|------|----------|
| `P2PMessageService.java` | process() 方法中多分支逻辑 | **管道模式**：限流→校验→去重→存储→推送→回调 |
| `GroupMessageService.java` | process() 方法中多分支逻辑 | **管道模式** |

| 场景 | 当前实现 | 建议 |

| 好友请求 | if-else 判断 | 状态机：PENDING→APPROVED/REJECTED |
| 群成员状态 | 多个字段判断 | 状态机：NORMAL→MUTED→REMOVED |

## 六、与 Vellastra 项目的关系

当前 Vela（IM 系统）和 Vellastra（内容博客系统）是两个独立项目：

| 维度 | Vela (IM) | Vellastra (Content) |
|------|-----------|---------------------|
| 定位 | 即时通讯引擎 | 内容管理系统 |
| 架构 | DDD 六边形 | DDD 四层 |
| 技术栈 | Netty/RabbitMQ/ZK/Redis | SpringBoot/MyBatis/MySQL |
| 成熟度 | 功能完整，缺深度 | 功能完整，含测试覆盖 |
| 建议 | 先补深度，再扩广度 | 保持现状，按需迭代 |

**不推荐强行合并**，两个系统关注点不同。但可以复用：
- `im-common` 中的基础工具类
- 用户认证体系（统一登录）
- 部署运维基础设施

## 附录：各模块 L2 异常检查清单（可直接使用）

### 消息发送

```
□ 网络超时：重试 3 次（指数退避：100ms → 500ms → 2s）
□ 服务端返回错误：展示错误原因，不重试
□ 客户端离线：转离线消息，上线后同步
□ 消息体过大：限制大小，提示用户
□ 发送频率过快：限流，提示"发送太频繁"
□ 对方已删除好友：提示"对方已不是好友"
□ 发送给自己：提示"不能给自己发消息"
□ 消息 ID 重复：幂等处理，返回已发送成功
```

### 消息撤回

```
□ 超时（>120s）：提示"已超过撤回时间"
□ 消息不存在：提示"消息不存在"
□ 已被撤回：提示"消息已被撤回"
□ 不是发送者：提示"只能撤回自己的消息"
□ 群聊中撤回：群成员都收到撤回通知
□ 离线用户撤回上线后：同步撤回状态
□ 弱网丢包：撤回失败，重试机制
□ 服务端时间与客户端时间差：以服务端为准
```

### 离线消息同步

```
□ 同步量过大：分批拉取（每次最多 100 条）
□ 重复同步：幂等处理（按 sequence 去重）
□ 拉取后服务端新增：重新拉取
□ 同步超时：重试，断点续传
□ Redis 数据丢失：从 DB 恢复
□ 同时多端拉取：每个端独立拉取，互不干扰
```
